## 2. Найденные медленные запросы (до оптимизации)
### Запрос №1
История заказов самого активного пользователя.

```sql
WITH busiest_user AS (
    SELECT user_id
    FROM orders
    GROUP BY user_id
    ORDER BY count(*) DESC, user_id
    LIMIT 1
)
SELECT
    id,
    user_id,
    status,
    total_amount,
    created_at
FROM orders
WHERE user_id = (SELECT user_id FROM busiest_user)
ORDER BY created_at DESC
LIMIT 20;
```

Ключевые узлы плана `EXPLAIN ANALYZE` до оптимизации:
- `Seq Scan on orders orders_1` -> `HashAggregate` -> `Sort` внутри `busiest_user`;
- повторный `Seq Scan on orders` для поиска заказов выбранного пользователя;
- `Sort` по `created_at DESC`.

`Execution Time`: `24.229 ms`

Почему запрос медленный:
- таблица `orders` читается полностью два раза;
- итоговый результат маленький (`20` строк), но чтение идёт по всем `100 000` заказам;
- сортировка выполняется после полного чтения отфильтрованного набора.

### Запрос №2
Оплата заказов по статусу и диапазону дат.

```sql
SELECT
    id,
    user_id,
    total_amount,
    created_at
FROM orders
WHERE status = 'paid'
  AND created_at >= TIMESTAMPTZ '2025-01-01 00:00:00+00'
  AND created_at < TIMESTAMPTZ '2025-04-01 00:00:00+00'
ORDER BY created_at DESC
LIMIT 100;
```

Ключевые узлы плана `EXPLAIN ANALYZE` до оптимизации:
- `Seq Scan on orders`;
- `Sort` с `top-N heapsort`.

`Execution Time`: `8.265 ms`

Почему запрос медленный:
- отбирается только часть таблицы (`6149` строк), но читаются все `100 000` заказов;
- нужный порядок по дате отсутствует в физической структуре, поэтому выполняется дополнительная сортировка.

### Запрос №3
Агрегация по выручке с `JOIN orders + order_items`.

```sql
SELECT
    oi.product_name,
    count(*) AS items_count,
    round(sum(oi.subtotal)::numeric, 2) AS revenue
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.created_at >= TIMESTAMPTZ '2025-07-01 00:00:00+00'
  AND o.created_at < TIMESTAMPTZ '2025-10-01 00:00:00+00'
GROUP BY oi.product_name
ORDER BY revenue DESC
LIMIT 20;
```

Ключевые узлы плана `EXPLAIN ANALYZE` до оптимизации:
- `Seq Scan on orders o`;
- `Seq Scan on order_items oi`;
- `Hash Join`;
- `HashAggregate`;
- `Sort` для `ORDER BY revenue DESC`.

`Execution Time`: `66.982 ms`

Почему запрос медленный:
- читаются все `300 000` строк `order_items`;
- объединение и агрегация выполняются на большом промежуточном наборе;
- без индекса по дате на `orders` фильтрация диапазона не может быть дешёвой.

### Запрос №4
Глобальный агрегат по месяцам и статусам.

```sql
SELECT
    date_trunc('month', created_at) AS month_bucket,
    status,
    count(*) AS orders_count,
    round(sum(total_amount)::numeric, 2) AS revenue
FROM orders
GROUP BY 1, 2
ORDER BY 1, 2;
```

Ключевые узлы плана `EXPLAIN ANALYZE` до оптимизации:
- `Seq Scan on orders`;
- `Sort`;
- `GroupAggregate`.

`Execution Time`: `97.581 ms`

Почему запрос медленный:
- запрос обрабатывает всю таблицу целиком;
- для группировки по месяцу и статусу PostgreSQL сортирует все `100 000` строк;
- индексами такой сценарий ускоряется слабо, потому что нужно прочитать почти все записи.

## 3. Добавленные индексы и обоснование типа
### Индекс №1
- SQL:
```sql
CREATE INDEX IF NOT EXISTS idx_orders_user_created_at_desc
    ON orders USING BTREE (user_id, created_at DESC);
```
- Какой запрос ускоряет: запрос №1.
- Почему выбран тип: `BTREE` хорошо работает для точного равенства по `user_id` и одновременно поддерживает упорядочивание по `created_at DESC` внутри каждой группы пользователя.

### Индекс №2
- SQL:
```sql
CREATE INDEX IF NOT EXISTS idx_orders_paid_created_at_desc
    ON orders USING BTREE (created_at DESC)
    INCLUDE (id, user_id, total_amount)
    WHERE status = 'paid';
```
- Какой запрос ускоряет: запрос №2.
- Почему выбран тип: выбран частичный покрывающий `BTREE`. Частичный индекс хранит только `paid`-заказы, а `INCLUDE` даёт возможность выполнить `Index Only Scan` без чтения heap-страниц для `id`, `user_id` и `total_amount`.

### Индекс №3
- SQL:
```sql
CREATE INDEX IF NOT EXISTS idx_orders_created_at_id
    ON orders USING BTREE (created_at, id);
```
- Какой запрос ускоряет: запрос №3.
- Почему выбран тип: `BTREE` подходит для диапазонного условия по `created_at`; добавление `id` помогает сразу получить ключи заказов для последующего `JOIN`.

### Индекс №4
- SQL:
```sql
CREATE INDEX IF NOT EXISTS idx_order_items_order_id
    ON order_items USING BTREE (order_id);
```
- Какой запрос ускоряет: запрос №3 и любые точечные выборки `order_items` по заказу.
- Почему выбран тип: `BTREE` является базовым и наиболее эффективным типом для соединения по равенству `oi.order_id = o.id`. В данном наборе данных планировщик не всегда выбирает этот индекс, потому что в агрегации участвует большая доля таблицы `order_items`.

## 5. Партиционирование `orders` по дате
### 5.1 Выбранная стратегия
Выбрана стратегия `RANGE PARTITION BY created_at` по кварталам:
- `2024_q1` ... `2024_q4`;
- `2025_q1` ... `2025_q4`;
- `DEFAULT`-партиция для выбросов вне ожидаемого диапазона.

Квартальные партиции удобны для запросов по временным диапазонам и при этом не создают слишком много отдельных таблиц.

### 5.2 Реализация
Реализация выполнена через `shadow-table` `orders_partitioned`, а не через замену исходной `orders`, чтобы не ломать внешние ключи и существующую OLTP-схему.

Сделано следующее:
- создана новая таблица `orders_partitioned` с `PARTITION BY RANGE (created_at)`;
- созданы 8 квартальных партиций и одна `DEFAULT`-партиция;
- перенесены все `100 000` строк из `orders` в `orders_partitioned`;
- на партиционированной таблице созданы аналоги основных индексов;
- выполнен `ANALYZE orders_partitioned`.

Распределение данных по кварталам получилось почти равномерным: примерно от `12.3k` до `12.6k` заказов на партицию, в `DEFAULT` попало `9` строк.

### 5.3 Проверка эффекта
После партиционирования эффект оказался неоднородным:
- `Q2` стал лучшим кейсом: сработало partition pruning, и PostgreSQL обратился только к `orders_partitioned_2025_q1`, после чего выполнил локальный `Index Only Scan`;
- `Q3` тоже получил pruning до одной партиции `orders_partitioned_2025_q3`, но основное время осталось в `Seq Scan on order_items`, `Hash Join` и агрегации;
- `Q4` не выиграл, потому что агрегирует все месяцы сразу, и план превратился в `Append` по всем партициям с последующей сортировкой и `GroupAggregate`;
- `Q1` почти не ускорился, так как подзапрос `busiest_user` по-прежнему должен агрегировать все заказы по всем партициям.

## 7. Что удалось исправить
- Убрано полное сканирование внешней части `Q1`: доступ к заказам пользователя теперь идёт через индекс `(user_id, created_at DESC)`.
- `Q2` переведён с полного сканирования таблицы на частичный покрывающий индекс, из-за чего запрос выполняется через `Index Only Scan`.
- Для `Q3` улучшена фильтрация по временному диапазону через индекс `(created_at, id)`, что уменьшило стоимость доступа к таблице `orders`.
- Реализовано партиционирование по дате без нарушения исходной схемы и внешних ключей за счёт отдельной таблицы `orders_partitioned`.
- Для диапазонных запросов по дате получено реальное pruning партиций, особенно заметное на `Q2`.

## 8. Что не удалось исправить только индексами
- `Q4` не удаётся эффективно ускорить только индексами, потому что запрос читает почти всю таблицу и считает агрегаты по всему диапазону. Здесь нужны materialized view, rollup-таблицы или предварительная агрегация.
- `Q3` ограниченно реагирует на индексацию, потому что основной объём работы находится в `Seq Scan` по `order_items` (`300 000` строк), `Hash Join` и `HashAggregate`. Для заметного ускорения потребовались бы переписывание запроса, отдельная агрегационная витрина или партиционирование/оптимизация `order_items`.
- `Q1` всё ещё содержит дорогое вычисление `busiest_user`, которое делает глобальную агрегацию по всем заказам. Индекс ускоряет только внешнюю выборку заказов уже найденного пользователя, но не убирает полную агрегацию.
- Индекс `idx_order_items_order_id` логически корректен, но в текущем профиле нагрузки не дал заметного эффекта, потому что планировщик рационально предпочёл `Seq Scan` по `order_items`: в запросе участвует слишком большая доля строк.

## 9. Выводы
1. `EXPLAIN ANALYZE` нужно снимать на реалистичном объёме данных: многие проблемы не видны на маленьких таблицах.
2. Наиболее сильный эффект дают индексы, точно совпадающие с шаблоном запроса: фильтр, сортировка и нужные столбцы должны учитываться вместе.
3. Частичный покрывающий индекс особенно эффективен для селективных условий наподобие `status = 'paid'`.
4. Партиционирование полезно только тогда, когда запрос реально может отбросить лишние партиции; для глобальных агрегатов по всему диапазону оно помогает мало или даже ухудшает ситуацию.
5. Не все проблемы лечатся индексами: тяжёлые `JOIN + GROUP BY` и полные агрегаты требуют изменения архитектуры запросов, pre-aggregation или дополнительных витрин.
